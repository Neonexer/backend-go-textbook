# Мониторинг и SLO

Метрики собираются. Что с ними делать? SLO переводит «сервис работает нормально» в измеримые цифры. Алерты срабатывают когда SLO под угрозой.

## SLI → SLO → SLA

| Аббревиатура | Значение | Пример |
|-------------|---------|--------|
| **SLI** (Indicator) | Что измеряем | Время ответа, доля ошибок |
| **SLO** (Objective) | Целевое значение | 99.9% запросов быстрее 300ms за 30 дней |
| **SLA** (Agreement) | Договор с последствиями | Если `&lt;99.5%` — штраф |

## Золотые сигналы (Google SRE)

| Сигнал | SLI |
|--------|-----|
| **Latency** | p50, p95, p99 времени ответа |
| **Traffic** | Запросов/сек |
| **Errors** | Доля 5xx |
| **Saturation** | Загрузка CPU, память, goroutines |

## Error budget

SLO 99.9% означает что за 30 дней (~43 000 минут) допустимо 43 минуты недоступности. Это **error budget**:

- Бюджет цел → можно деплоить рискованные изменения
- Бюджет исчерпан → стоп деплои, фиксим стабильность

## Алерты

Не алерти на каждый 500. Алерти на **скорость расхода error budget**:

```
Если за 1 час израсходовано 20% месячного error budget → алерт
Если за 6 часов израсходовано 50% → критический алерт
```
