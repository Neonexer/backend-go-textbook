# Введение в системный дизайн

Ты умеешь писать REST API и микросервисы. Теперь вопрос масштаба: что делать когда пользователей не 100, а 100 000 000? Системный дизайн отвечает на этот вопрос.

## О чём этот раздел

Системный дизайн — не код, а архитектурные решения для масштабируемых систем. Мы пройдём ключевые паттерны:

- **Load Balancing** — распределение трафика между серверами
- **Sharding** — распределение данных между серверами БД
- **CQRS** — разделение чтения и записи
- **Saga** — распределённые транзакции в микросервисах
- **Event Sourcing** — журнал событий как источник истины

## Вертикальное vs горизонтальное масштабирование

**Вертикальное** — купить сервер мощнее. Быстро, дорого, есть предел (самый мощный сервер всё ещё один).

**Горизонтальное** — добавить серверов. Сложнее (нужен балансировщик, шардинг), но масштабируется линейно.

Все паттерны в этом разделе — про горизонтальное масштабирование.

## CAP-теорема

В распределённой системе из трёх свойств можно выбрать только два:

- **C**onsistency — все узлы видят одинаковые данные
- **A**vailability — каждый запрос получает ответ (не ошибку)
- **P**artition tolerance — система работает при разрыве сети

:::tip Выбор на практике
При partition (разрыв сети) ты выбираешь между C и A. Большинство интернет-компаний выбирают AP (доступность важнее консистентности). Банки — CP (консистентность важнее).
:::

## Ключевые вопросы при проектировании

1. Сколько пользователей? (1000? 100М?)
2. Какая нагрузка: read-heavy или write-heavy?
3. Какая задержка допустима? (100ms? 10s?)
4. Какая консистентность нужна? (строгая? eventual?)
