# CQRS — разделение команд и запросов

В типичном CRUD приложении одна и та же модель и для записи, и для чтения. Но часто пишем мы редко, а читаем часто и в разных разрезах. CQRS разделяет модель на две.

## Идея

```
         ┌──────────┐
  POST   │ Command  │────▶ БД (write-оптимизированная)
  PUT    │  Model   │
  DELETE └──────────┘
              │
              │ (события)
              ▼
         ┌──────────┐
  GET    │  Query   │────▶ Read-оптимизированная БД
         │  Model   │       (денормализованная, кешированная)
         └──────────┘
```

## Пример: YouTube-клон

**Write side** (команды):
- Загрузить видео → таблица `videos` (нормализованная)
- Поставить лайк → таблица `likes`

**Read side** (запросы):
- Страница видео: название + автор + лайки + первые 10 комментариев
- Вместо 5 JOIN'ов — одна денормализованная таблица `video_views`

## Когда CQRS

- ✅ Разные паттерны доступа на чтение и запись
- ✅ Чтений в 100× больше чем записей
- ✅ Сложные JOIN'ы для отображения
- ❌ Простой CRUD (избыточно)
