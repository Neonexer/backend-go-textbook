# Шардинг баз данных

Один сервер PostgreSQL держит 1TB данных и 5000 запросов/сек. Когда этого мало — нужно распределить данные между несколькими серверами. Это шардинг.

## Горизонтальный шардинг

Разделяем таблицу на части (шарды) по ключу:

```
Шард 0: пользователи A–M  (10.0.0.1)
Шард 1: пользователи N–Z  (10.0.0.2)
Шард 2: пользователи 0–9  (10.0.0.3)
```

## Стратегии выбора шарда

### Range-based

```go
func shardForUser(userID int) int {
    return userID % numberOfShards
}
```

Просто. Проблема: нельзя добавить шард без перераспределения ВСЕХ данных.

### Hash-based (Consistent Hashing)

Серверы и ключи хешируются на кольцо. Добавление нового шарда затрагивает только соседние:

```
Кольцо:  0 ── A ── 100 ── B ── 200 ── C ── 256
           ↑       ↑        ↑        ↑
        server1  server2  server3  server1 (повтор)
```

Добавили server4 между A и B — переезжает только диапазон A→100.

### Directory-based

Отдельный сервис хранит маппинг: `{user_id: 42 → shard: 3}`. Гибко, но ещё одна зависимость.

## Проблемы шардинга

- **JOIN между шардами** — нельзя. JOIN с данными другого шарда невозможен.
- **Распределённые транзакции** — данные на разных шардах требуют two-phase commit или Saga.
- **Ребалансировка** — при добавлении шарда нужно перемещать данные.
