# Микросервисная архитектура

В Проектах 1 и 2 всё приложение жило в одном процессе — **монолит**. Для YouTube-клона это не подойдёт: загрузка видео, транскодирование, комментарии, уведомления — всё в одном процессе не масштабируется. Разделим на микросервисы.

## Почему микросервисы для YouTube-клона

У разных частей системы разная нагрузка:

- **Video Service** — загрузка и хранение видео, самое нагруженное
- **Transcoder** — CPU-тяжёлая задача, должна масштабироваться отдельно
- **Comment Service** — много записей, мало нагрузки
- **Notification Service** — может простаивать часами

В монолите все они делят один процесс. Если Transcoder загрузит CPU — остальные сервисы встанут.

## Сервисы YouTube-клона

```
┌──────────┐     ┌───────────┐     ┌──────────┐
│  API     │────▶│  Video    │────▶│  S3      │
│  Gateway │     │  Service  │     │  (файлы) │
└────┬─────┘     └─────┬─────┘     └──────────┘
     │                 │
     │           ┌─────▼─────┐
     │           │   Kafka   │──▶ Transcoder
     │           └─────┬─────┘
     │                 │
     ├─────────▶ Comment Service
     ├─────────▶ User Service
     └─────────▶ Notification Service
```

| Сервис | Ответственность | Протокол |
|--------|----------------|---------|
| API Gateway | Единая точка входа, роутинг | HTTP (внешний) |
| Video Service | Загрузка, метаданные, стриминг | gRPC |
| Comment Service | Комментарии к видео | gRPC |
| User Service | Пользователи, подписки | gRPC |
| Transcoder | Обработка видео | Kafka consumer |
| Notification | Email, push | Kafka consumer |

## Взаимодействие: синхронное и асинхронное

**Синхронное** (gRPC) — когда клиент ждёт ответ:
- «Дай метаданные видео #42»
- «Кто подписан на пользователя?»

**Асинхронное** (Kafka) — когда ответ не нужен сразу:
- «Видео загружено — начни транскодирование»
- «Новый комментарий — отправь уведомление автору видео»

Правило: если вызывающий **не ждёт** результат — Kafka. Если ждёт — gRPC.

## Структура Go-проекта для микросервиса

Каждый сервис — отдельный Go-модуль с общей структурой:

```
video-service/
├── cmd/server/main.go
├── internal/
│   ├── handler/      # gRPC handlers
│   ├── service/      # бизнес-логика
│   └── repository/   # БД / S3
├── api/
│   └── video.proto   # protobuf-спецификация
└── go.mod
```

## Ключевые выводы

- Микросервисы — не серебряная пуля. Начинай с монолита, разделяй по мере роста.
- Синхронное общение (gRPC) — когда нужен ответ. Асинхронное (Kafka) — когда не нужен.
- Каждый сервис — свой модуль, своя БД, свой деплой.
