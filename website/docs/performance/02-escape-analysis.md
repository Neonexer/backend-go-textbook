# Escape Analysis

Почему бенчмарк показывает 0 allocs/op у одной функции и 5 allocs/op у другой? Escape analysis решает: останется переменная на стеке или уйдёт в кучу.

## Стек vs Куча

| | Стек | Куча |
|---|---|---|
| Скорость | Мгновенно (pointer bump) | Медленно (поиск + GC) |
| Время жизни | Пока функция не вернётся | Пока GC не соберёт |
| Размер | Ограничен (~1KB на фрейм) | Без ограничений |

## Когда значение убегает (escapes)

```go
func newPost() *Post {
    p := Post{Title: "test"} // p на стеке?
    return &p                 // указатель покидает функцию → p в кучу
}
```

```bash
go build -gcflags="-m"
# ./main.go:5:2: &p escapes to heap
```

## Проверка escape analysis

```bash
go build -gcflags="-m -m"  # два -m для деталей
```

Вывод подскажет:
- `does not escape` — на стеке ✅
- `escapes to heap` — в куче
- `moved to heap` — компилятор перенёс

## Практические советы

- **Не бойся кучи**. GC в Go быстрый. Преждевременная оптимизация — зло.
- **Профилируй сначала**. pprof показывает реальную проблему, а не гипотетическую.
- **Часто вызываемые функции** — кандидаты на 0 allocs/op (горячий путь).
- **`sync.Pool`** — если аллокации неизбежны, переиспользуй объекты.
