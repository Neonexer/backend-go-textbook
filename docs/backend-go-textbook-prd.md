**Автор**: Sarah (Product Owner)
**Оценка качества**: 92/100

## Executive Summary

Цифровой учебник «Бэкенд на Go» — это статический веб-сайт на Docusaurus, который проводит опытного разработчика через три сквозных проекта по нарастающей сложности: от простого REST API до полноценного YouTube-клона в микросервисной архитектуре.

Целевая аудитория — разработчики с опытом на Python, Java, JavaScript и других языках, которым нужен структурированный и практический путь в Go-бэкенд. Учебник покрывает полный производственный цикл: написание кода, тестирование, базы данных, брокеры сообщений, деплой и observability.

Главный критерий успеха: после прохождения учебника выпускник способен самостоятельно спроектировать и реализовать бэкенд масштаба YouTube-клона.

---

## Problem Statement

**Текущая ситуация**: Существующие ресурсы по Go либо объясняют синтаксис без привязки к реальным проектам, либо фокусируются на узких темах без целостной картины. Опытный разработчик вынужден собирать знания по кусочкам из документации, блогов и видео, теряя время и контекст.

**Предлагаемое решение**: Единый структурированный учебник, построенный вокруг трёх проектов. Каждая глава — это конкретный шаг в проекте с объяснением концепций, примерами кода и проверочными тестами. Читатель не просто читает — он пишет работающий код.

**Бизнес-импакт**: Сокращение времени T-shaped-освоения Go-бэкенда с месяцев до недель. Учебник становится референсным ресурсом для сообщества.

## Success Metrics

**Первичные KPI:**
- **Готовность к реальному проекту**: выпускник учебника способен самостоятельно написать бэкенд YouTube-клона (архитектура + код + деплой)
- **Покрытие тем**: 100% ключевых тем production-бэкенда покрыты в программе (REST API, БД, аутентификация, микросервисы, брокеры, observability, CI/CD)
- **Качество контента**: каждый пример кода компилируется и проходит тесты; каждый тест к главе имеет однозначно правильный ответ

**Валидация**: на этапе написания — каждый сниппет проверяется компилятором Go и линтером. После публикации — обратная связь от сообщества (issues/PR на GitHub).

### Основная: Опытный разработчик, осваивающий Go

- **Роль**: Backend/Frontend-разработчик с 2+ годами опыта на Python, Java, JS, C# или PHP
- **Цели**: быстро освоить Go для бэкенд-разработки на продакшен-уровне; получить практический опыт через реальные проекты
- **Болевые точки**: существующие материалы слишком базовые или слишком фрагментированные; нет единого пути от нуля до production-grade

## User Stories & Acceptance Criteria

### Story 1: Последовательное обучение через проекты

**Как** опытный разработчик
**Я хочу** проходить главы последовательно, строя проекты шаг за шагом
**Чтобы** к концу учебника иметь портфолио из трёх работающих проектов и глубокое понимание Go-бэкенда

**Acceptance Criteria:**
- [ ] Все главы выстроены в логическом порядке, каждая опирается на предыдущие
- [ ] Код каждого проекта компилируется и запускается независимо
- [ ] Перекрёстные ссылки между главами работают корректно
- [ ] Читатель видит свой прогресс по главам (опционально, через LocalStorage)

### Story 2: Проверка знаний через тесты

**Как** читатель
**Я хочу** после каждой главы проходить тест для самопроверки
**Чтобы** убедиться, что я усвоил материал перед переходом к следующей теме

**Acceptance Criteria:**
- [ ] Каждая глава завершается интерактивным multiple-choice тестом (3-7 вопросов)
- [ ] Тест показывает правильные/неправильные ответы сразу после выбора
- [ ] Тест показывает итоговый результат (например, «4/5»)
- [ ] Пройденные тесты сохраняются в LocalStorage браузера

### Story 3: Работа с примерами кода

**Как** читатель
**Я хочу** копировать примеры кода одним кликом и видеть читабельную подсветку синтаксиса
**Чтобы** быстро вставлять код в свой редактор и экспериментировать локально

**Acceptance Criteria:**
- [ ] Все блоки кода имеют кнопку «Скопировать»
- [ ] Подсветка синтаксиса Go корректна и соответствует стандартам (Prism)
- [ ] Многострочные примеры сохраняют форматирование при копировании
- [ ] Ключевые сниппеты помечены ссылкой «открыть в Go Playground»

### Story 4: Навигация и поиск

**Как** читатель
**Я хочу** быстро находить нужную тему через поиск и навигационное меню
**Чтобы** использовать учебник как справочник после прохождения

**Acceptance Criteria:**
- [ ] Полнотекстовый поиск по всем главам
- [ ] Боковое меню с древовидной структурой глав
- [ ] Хлебные крошки на каждой странице
- [ ] Предыдущая/следующая глава внизу страницы

## Functional Requirements

### Core Features

**Feature 1: Контент — три сквозных проекта**

Проект 1 — «REST API для блога» (основы Go и HTTP):
- `net/http` и роутер `chi`: маршрутизация, параметры URL, группы маршрутов
- Middleware-паттерн: логирование, аутентификация, recovery, CORS
- JSON-сериализация и десериализация: `encoding/json`, кастомные marshal/unmarshal
- Валидация входящих данных (структурные теги + библиотека `validator`)
- Структура проекта: разделение на handler/service/repository
- Тестирование HTTP-обработчиков: `httptest`, табличные тесты, моки
- Структурное логирование с `log/slog`
- Graceful shutdown HTTP-сервера

Проект 2 — «Маркетплейс» (базы данных и аутентификация):
- SQL и ORM: `database/sql`, `pgx`, **GORM** — сравнение подходов, когда что выбирать
- Миграции: `golang-migrate`, версионирование схемы
- Аутентификация: JWT (access + refresh токены), middleware для защищённых эндпоинтов
- Авторизация: RBAC — роли и permission
- Транзакции и конкурентный доступ: `SELECT FOR UPDATE`, пессимистичная/оптимистичная блокировка
- Пагинация (cursor-based и offset-based), фильтрация, полнотекстовый поиск
- Загрузка файлов: multipart, валидация, хранение (локально / S3-совместимое)
- Интеграционные тесты с testcontainers

Проект 3 — «YouTube-клон» (микросервисы и продакшен):
- Микросервисная архитектура: разделение на сервисы (video, user, comment, notification), API Gateway
- gRPC: protobuf, кодогенерация, межсервисное общение
- Брокер сообщений: **Apache Kafka** — продюсеры, консьюмеры, consumer groups, exactly-once семантика
- Кеширование: Redis — стратегии кеширования, инвалидация, Cache-Aside
- Observability: OpenTelemetry (трассировка), Prometheus (метрики), Loki/ELK (логи)
- Rate limiting: token bucket на уровне middleware
- CI/CD: GitHub Actions — линтинг, тесты, сборка Docker-образов, деплой
- Деплой: Docker Compose (dev/staging), Kubernetes (production) — манифесты, Helm

**Feature 2: Интерактивные тесты**
- JavaScript-виджет, встроенный в MDX-страницу
- Multiple-choice с немедленной обратной связью
- Итоговый счёт по тесту
- Состояние «пройдено/не пройдено» в LocalStorage

**Feature 3: Подсветка кода и примеры**
- Prism-подсветка с темной и светлой темой
- Кнопка копирования на каждом блоке кода
- Ссылки на Go Playground для избранных сниппетов
- Все примеры кода проходят компиляцию (`go build`) перед публикацией

### Out of Scope
- Полноценный LMS (Learning Management System) с серверной частью
- Социальные функции: комментарии, форум, рейтинг
- Видео-лекции
- Интерактивный sandbox с запуском кода в браузере
- Адаптация под мобильные устройства как основная цель (достаточно читаемости)
- Платная подписка / paywall

## Technical Constraints

### Performance
- Lighthouse Score ≥ 95 (статический сайт — достигается автоматически с Docusaurus)
- Время загрузки страницы < 2 секунд (включая подсветку кода и тесты)
- Размер страницы < 500 KB (без учёта изображений)

### Security
- Статический сайт — минимальная поверхность атаки
- Нет пользовательских данных на сервере
- HTTPS enforced на уровне хостинг-платформы
- Зависимости Docusaurus/Node.js регулярно обновляются (Dependabot)

### Integration
- **Docusaurus**: генератор статического сайта, React/MDX
- **Prism**: подсветка синтаксиса (встроен в Docusaurus)
- **GitHub Pages / Vercel**: хостинг и деплой
- **Go Playground**: внешние ссылки для запуска примеров

### Technology Stack
- **Фреймворк**: Docusaurus 3.x
- **Язык**: TypeScript/JavaScript (конфигурация и кастомные компоненты)
- **Контент**: Markdown + MDX (для страниц с интерактивными тестами)
- **Тесты (виджет)**: vanilla JS или Preact (минимизация размера бандла)
- **Совместимость**: Chrome, Firefox, Safari, Edge последних 2 версий
- **Инфраструктура**: GitHub репозиторий, деплой через GitHub Actions

## Risk Assessment

| Риск | Вероятность | Влияние | Стратегия смягчения |
|------|------------|--------|---------------------|
| Примеры кода устаревают с новыми версиями Go | Средняя | Высокое | CI-проверка всех примеров на каждой новой версии Go; Dependabot для зависимостей |
| Слишком большой объём контента — потеря фокуса | Средняя | Среднее | Жёсткая структура: одна глава = одна тема = один шаг в проекте. Без отступлений. |
| Низкая вовлечённость (читатели бросают) | Высокая | Среднее | Тесты после каждой главы; видимый прогресс; проектная мотивация |
| Конкуренция с существующими ресурсами (Let's Go, Go by Example) | Средняя | Низкое | Уникальное преимущество: три сквозных проекта + production-цикл + русский язык |
| Kafka может быть overkill для примеров | Средняя | Низкое | Упрощённая локальная настройка через Docker Compose; объяснение «почему Kafka, а не RabbitMQ» |

## Appendix

### Глоссарий
- **MDX**: расширение Markdown, позволяющее встраивать JSX/React-компоненты прямо в текст

- **Kafka**: распределённый брокер сообщений с моделью pub/sub и персистентным хранением
